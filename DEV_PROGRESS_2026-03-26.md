# EasyChat for Edge - Progress 2026-03-26

## 今日追加完成

### 1. 来源徽章从展示升级为可操作

- `assistant` 回复下的来源徽章现在可点击
- 在 `popup / sidebar / full window` 三入口里行为一致
- 不再只是显示来源类型

当前行为：

- 有 `url` 的来源可展开详情并打开原页
- 有标题 / 摘录的来源可复制摘要
- 来源详情卡可展示标题、链接、摘录

相关文件：

- `shared-core.js`
- `popup.js`
- `chat.js`

### 2. 来源详情支持“定位来源”

- 来源详情卡新增“定位来源”按钮
- 优先复用已有 tab，再尝试新开来源页
- 会向来源页注入 `content.js` 并请求页内定位

当前行为：

- 找到文本时会滚动到对应位置并高亮
- 复用已有 tab 时不会重复开一堆页面
- `popup` 和 `full window` 现在都能触发同一套定位逻辑

相关文件：

- `content.js`
- `popup.js`
- `chat.js`

### 3. 页内高亮能力补强

- 来源高亮现在会跟随页面滚动和窗口尺寸变化刷新
- 高亮支持 `Esc` 关闭
- 高亮支持页面右上角关闭按钮

这部分解决过的问题：

- 高亮第一次定位后滚动页面会漂移
- 高亮只能等超时自动消失

相关文件：

- `content.js`

### 4. 定位匹配做了多层回退

现在的定位顺序：

1. 优先用来源 `preview`
2. 再尝试来源 `title`
3. 再尝试从摘录里拆出来的短片段
4. 最后走去标点 / 压空白的宽松匹配

这让这些情况命中率更高：

- 页面断句和原摘录不完全一致
- 标点不同
- 摘录被截短
- 输入框 / 编辑器内容存在轻微格式差异

### 5. 定位结果反馈更具体

定位成功后不再统一只提示“已定位到来源位置”，现在会区分：

- 按摘录定位
- 按标题定位
- 宽松匹配定位

相关文件：

- `content.js`
- `popup.js`
- `chat.js`

### 6. 联网搜索来源升级为真实结果级来源

- 搜索层现在不只返回拼好的文本，还会返回结构化来源
- 真实搜索结果会写进用户消息的 `meta.contextSources`
- 后续 assistant 回复会继承这些真实搜索来源

当前效果：

- 联网搜索回答下的来源徽章不再只是泛化的 `🌐 联网搜索`
- 现在会对应到真实搜索结果标题 / 链接 / 摘要
- 来源详情卡可直接对这些搜索结果展开、打开原页、尝试定位

相关文件：

- `shared-core.js`
- `popup.js`
- `chat.js`

## 下一步建议

优先继续：

1. 开始做“自动帮我操作浏览器”主链路
2. 做“定位失败”的第二层智能回退
3. 给来源详情卡增加最近一次定位结果提示

其中明天的第一优先级：

1. 先做浏览器操作基础能力：读取当前页、识别可点击元素、输入文字、点击、滚动
2. 再做自然语言动作执行：例如“帮我点登录”“帮我在这个输入框里填内容”
3. 最后把执行结果回传到聊天区，并补失败重试与安全限制

如果继续排查 bug，优先手测：

1. 普通文章页的来源定位
2. 输入框 / 富文本编辑器的来源定位
3. 长页面滚动后的高亮是否持续跟随
4. `popup -> 来源详情 -> 定位来源`
5. `full window -> 来源详情 -> 定位来源`
